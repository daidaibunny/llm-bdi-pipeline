\section{Method}
\label{sec:method}

\subsection{Problem Setting}

Let $D = \langle \mathcal{P}, \mathcal{A} \rangle$ be a typed \pddl{} domain
with predicate schemas $\mathcal{P}$ and action schemas $\mathcal{A}$. A
training split $\mathcal{I}_{train}$ contains planning problems over $D$. The
goal of the atomic component is to obtain a lifted atomic \asl{} library
$\mathcal{L}_D$ whose plan heads are declared predicate or literal goals, such
as \texttt{+!on(X,Y)} or \texttt{+!served(P)}. The library is domain-level:
training-object names must not appear in plan heads, and the library is reused
across future queries for the same domain.

A user query is handled by a temporal component. The Input component produces a
lifted LTLf JSON object. We parse that object, compile the LTLf formula into a
deterministic finite automaton, validate that relevant transition guards are
conjunctions of declared positive or negative literals, and append a
query-specific top-level \asl{} goal such as
\texttt{+!g\_query\_17}. The appended wrapper calls the atomic library; it does
not replace the atomic library.

\subsection{Atomic Template Evidence}

From the training split we extract the set of atomic goal templates:
\[
	\begin{aligned}
	T_D = \{(p,k,\sigma) \mid {}&p(o_1,\ldots,o_k) \text{ or }
	\neg p(o_1,\ldots,o_k) \\
	&\text{appears as a training goal item}\}.
	\end{aligned}
\]
where $\sigma$ records positive or negative polarity. These templates are the
interface to external generalized-planning providers. The project does not first
classify the domain into a provider-specific taxonomy. Instead, it asks which
provider can produce reusable lifted artifacts for the required atomic
templates.

The Evidence Module is the interface to external generalized-planning
providers. An Evidence Module adapter is a provider-specific parser that emits a
provider-neutral evidence program. For example, the MOOSE adapter reads a
\texttt{policy --dump-policy} decision-list rule whose goal condition is
\texttt{at(p0,l2)} and whose action sequence is a sequence of PDDL
operators such as \texttt{load}, \texttt{fly}, and \texttt{unload}. The
emitted evidence program contains the target
goal atom, context atoms, numeric guards when present, macro actions, and
provenance.

MOOSE is the first evidence provider we instantiate for positive singleton
predicate templates because its synthesis algorithm decomposes training
problems into singleton goal conditions and uses goal regression to learn
lifted rules \cite{Chen2026Moose}. This directly matches our
atomic-template interface. MOOSE is the only provider implemented and evaluated
in this work. The normalized interface permits a future provider to emit the
same evidence program and satisfy the same schema-level obligations, but we do
not claim empirical provider interchangeability.

Negative literal templates are not part of the current positive claim. If a
training split requires them, they require a provider-side lifted rule and a
compiler-side certificate that the generated module really achieves the
negative literal.

\subsection{Evidence Gate and Policy-Lifting Compiler}

Every consumed provider artifact is normalized before compilation:
\[
\begin{aligned}
	&\text{provider artifact} \rightarrow \text{Evidence Module adapter} \\
	&\quad\rightarrow \texttt{PolicyEvidenceProgram} \\
	&\quad\rightarrow \text{validated policy-lifting compiler}
	\rightarrow \text{\asl{} library}.
\end{aligned}
\]
Here a \texttt{PolicyEvidenceProgram} is the evidence representation between a
provider and the compiler: it is a normalized set of singleton-goal rules with
contexts, macro actions, numeric conditions, and provenance. The compiler then
checks that each candidate rule is compatible with the PDDL domain schema and
that the rendered \asl{} plan remains lifted. Provider-specific artifacts that
cannot satisfy this contract are not compiled into the domain library. This is
the main decoupling point: the Evidence Module decides how external planning
artifacts become evidence, while the compiler decides which evidence is safe to
render as reusable agent code.

The compiler constructs one candidate space containing validated provider
macros, direct PDDL producers, recursive preparation branches, certified
resource-release branches, and validated bounded-integer numeric branches.
Schema-derived primitive sequences come from a fixed, domain-independent
grammar: a direct producer; a support producer followed by the target producer;
one optional bridge producer between them; one optional prefix producer before
that bridge sequence; and one optional certified cleanup action. Consequently,
schema-derived candidates contain at most five primitive actions. Validated
provider macros are not truncated to this bound. The bound defines the searched
hypothesis class and is recorded in artifact metadata; it is not presented as a
MOOSE parameter or as completeness for arbitrary PDDL.
Predicate and numeric evidence therefore enter the same Clingo solve rather
than being selected and merged in separate phases. One Clingo program enforces evidence coverage and
internal-module closure, maximizes compatible certified recursive capabilities,
and then minimizes branch, context, and body cost. This use of answer-set
optimization follows the general synthesis pattern used for learned planning
sketches \cite{Drexler2022LearningSketches}, but our decision variables denote
executable AgentSpeak module branches rather than abstract sketch rules.
The resulting optimum is relative to the generated certified candidate space;
we do not claim optimality over all AgentSpeak programs or completeness for
arbitrary PDDL domains.

Every selected recursive branch requires a compile-time well-founded feature.
The fixed, domain-independent feature grammar currently contains (i) the
global number of atoms of a dynamic relation and (ii) the number of relation
atoms in the acyclic obstruction cone rooted at a query argument. The second
feature permits a producer to relocate an obstruction while preserving the
global atom count, but only when schema guards prove that the new relation
value differs from the protected root. A recursive candidate is compatible
only with selected branches whose reachable module closure does not increase
its selected feature, and at least one certified producer sequence must
strictly decrease it.
The feature is metadata and is not emitted as an AgentSpeak belief. This design
is related to feature-defined general policies \cite{Frances2021LearningGeneralPolicies},
while restricting the claim to features whose changes can be certified from
the PDDL schemas.

A resource cleanup requires a keyed capacity invariant. The producer must
consume a key-only free mode and create a key-plus-occupant debt mode; the
cleanup must perform the inverse transition while preserving the atomic target.
Structurally symmetric modes whose direction cannot be inferred are rejected.
Thus resource roles are derived from paired causal effects rather than fluent
names.

\subsection{Temporal Goal Append}

The temporal layer consumes a lifted LTLf goal case:
\[
	\langle q, g_q, \varphi_q, A_q, B_q \rangle,
\]
where $q$ is a query identifier, $g_q$ is the top-level \asl{} goal name,
$\varphi_q$ is an LTLf formula, $A_q$ lists declared atoms, and $B_q$ is an
optional variable-to-object binding map used by the external Input component.

The LTLf formula is compiled to a deterministic finite automaton by the real
\texttt{ltlf2dfa}/MONA toolchain. Every progress edge on the unique accepting
path becomes one query-local transition helper. A guard may contain conjunction
and negation but not disjunction. Accepting self-loops labelled \texttt{true}
are automaton plumbing and do not become atomic subgoals.

For one positive literal $p(\bar{X})$, the helper has an already-satisfied
branch and a repair branch that calls $p$ once and rechecks the same transition.
For a conjunction, the compiler computes conservative conditional may-delete
summaries over the final selected atomic module call graph. Each delete retains
the positive, negative, equality, and disequality context of its source branch.
The summary records net effects at successful atomic-module completion, so a
fact deleted and restored within one primitive macro is not treated as a final
delete. It computes a relational fixed point over reachable module calls:
query variables remain symbolic anchors, internal variables are
alpha-normalized, PDDL type constraints restrict unification, and
schema-derived single-valued invariants reject inconsistent conjunctive goals.
This covers parameter-changing recursion without a benchmark-specific depth
bound and preserves shared-variable identity across guard literals.
An edge $G_j \rightarrow G_i$
means that achieving $G_j$ may delete $G_i$, so $G_j$ must be repaired first.
An acyclic graph is topologically serialized. One cyclic structural case is
admitted by an assumption-bounded support-depth certificate: all positive
literals must use the same certified binary relation, their requested relation
graph must be functional and acyclic, and the selected library must contain a
well-founded decrease certificate for that relation. The resulting order builds
supports before dependants. The child/support argument orientation is derived
from the recursive certificate; the compiler also verifies that the recursive
module closure does not re-add the relation and that primitive relation
producers delete only the same child's previous relation. This certificate assumes that the relation remains
acyclic in every reachable execution state; it records that assumption rather
than claiming to derive it from arbitrary STRIPS schemas. Other incomplete or
cyclic graphs are rejected.
For example, requested atoms $r(a,b)$ and $r(b,c)$ are ordered as $r(b,c)$
before $r(a,b)$. A reachable state containing both $r(a,b)$ and $r(b,a)$ is
outside the assumption because recursive support depth would no longer be a
well-founded ranking.
This reordering is confined to literals on one DFA edge and therefore does not
change temporal order between edges. Negative literals remain context guards
and are never compiled as negative achievements.

A singleton numeric transition such as $fuel(v)=0$ is supported when its
selected module has a certified unit monotone effect. A mixed or multi-literal
transition such as $at(p,d) \land fuel(v)=3$ is rejected: the temporal compiler
does not yet prove that repairing one conjunct preserves the other's numeric
value.

The completion-level observation boundary is an explicit scope restriction.
It is sound for the achievement-transition fragment evaluated here, where a
transition guard is tested after an atomic subgoal returns. A formula whose
safety semantics must inspect every primitive intermediate action requires an
external DFA monitor and is outside this context-only wrapper claim.
For example, a module may temporarily delete and then restore a fluent before
returning; this is acceptable for an eventual achievement checked at return,
but cannot establish a safety formula such as $G\,safe$ without observing
every primitive successor state.

\subsection{Controller State}

A pure context-only \asl{} wrapper is not sufficient for every deterministic
finite automaton. If two automaton states expose the same literal or if temporal
progress cannot be inferred from the current world state, an external
reward-machine state must be maintained by the temporal controller. This is a
correctness boundary of the method.

\subsection{Selected Evaluation Scope}

The benchmark scope is used to evaluate coverage of literature-grounded domain
properties, not to route evidence providers by class:

\begin{table}[t]
	\caption{Current benchmark evaluation scope.}
	\label{tab:benchmark-scope}
	\small
	\begin{tabular}{p{0.33\columnwidth}p{0.43\columnwidth}}
		\toprule
		Group & Domains \\
		\midrule
		ESHO classical domains & Barman, Ferry, Gripper, Logistics, Miconic, Rovers, Satellite, and Transport \\
		Numeric fluent domains & Numeric Ferry, Numeric Miconic, Numeric Minecraft, and Numeric Transport \\
		Feature-definable serialized-width domains & Blocksworld Clear, Blocksworld On, Blocksworld Tower, and Depots \\
		\bottomrule
	\end{tabular}
\end{table}

The benchmark unit is a planning-family entry, not necessarily a unique action
schema file. Blocksworld Clear and Blocksworld On follow the standard
\texttt{QClear} and \texttt{QOn} singleton-goal families, while Blocksworld
Tower represents the classical support-dependent arrangement family. This lets
the evaluation separate singleton literal modules such as \texttt{clear(X)}
and \texttt{on(X,Y)} from tower requests whose support literals must be
serialized.

Numeric domains are included when the evidence provider supplies validated
numeric fluent rules and the compiler can preserve their resource semantics in
\asl{}.
