\section{Method}
\label{sec:method}

\subsection{Problem Setting}

Let $D = \langle \mathcal{P}, \mathcal{A} \rangle$ be a typed \pddl{} domain
with predicate schemas $\mathcal{P}$ and action schemas $\mathcal{A}$. A training
set $\mathcal{I}_{train}$ contains small planning problems over $D$, each with an
initial state and a positive conjunctive goal. A counterexample set
$\mathcal{I}_{cex}$ is initially empty and grows when the current library fails
on a held-out problem in a way that can be lifted.

The target artifact is a finite set $\mathcal{L}$ of lifted \asl{} plan schemas.
The library is domain-level: variables may appear in plan heads, contexts, and
bodies, but concrete object names from training problems must not appear in plan
heads. The only allowed achievement-goal heads are declared \pddl{} predicates
and the distinguished composer goal \texttt{g}. The body of a plan may call a
declared primitive action or another declared predicate achievement goal.

\subsection{Goal-Conditioned Representation}

The library observes three kinds of context facts. State facts are ordinary
\pddl{} fluents. Goal facts are read-only descriptors derived from the current
problem goal. For every predicate $p \in \mathcal{P}$, a goal literal
$p(o_1,\ldots,o_k)$ is represented as $\goalpred{p}(o_1,\ldots,o_k)$. These
facts are not actions, mutable beliefs, or achievement-goal heads. They are
inputs that let a single domain-level library distinguish one problem goal from
another. Ready facts are also read-only, but they are derived at run time from
the selected Layer C support agenda, the current state, and the goal facts.
A context atom \texttt{ready\_$p$(X)} means that no selected support-agenda
predecessor for \texttt{goal\_$p$(X)} remains unsatisfied. Ready facts gate
generic composer rules; they are not \pddl{} predicates, mutable beliefs,
primitive actions, or achievement-goal heads.

A context is a conjunction of state literals, negation-as-absence literals, type
guards, equality or inequality tests, read-only goal descriptors, and read-only
ready facts. The method uses only symbols declared by the \pddl{} domain, except
for the systematic read-only prefixes used for goal descriptors and ready
contexts.

\subsection{Architecture Overview}

The synthesis pipeline has three logical stages. First, the system constructs a
schema model from \pddl{} predicates, actions, types, preconditions, add
effects, and delete effects. Second, it generates candidate lifted rules for two
learned layers. Layer B candidates implement atomic predicate goals. Layer C
candidates compose conjunctive goals by calling Layer B modules under
goal-conditioned contexts. Third, an answer-set selector chooses a safe and
small subset of these candidates, and the compiler renders the selected rules as
\asl{} plans.

\subsection{Hypothesis Class}

The search space contains two families of lifted rules.

Layer B contains atomic predicate-goal modules. A rule has the form
\[
	+!p(\bar{X}) : C(\bar{X},\bar{Y}) \leftarrow B_1;\ldots;B_m
\]
where $p \in \mathcal{P}$, $C$ is a safe lifted context, and each body item is
either $a(\bar{Z})$ for some $a \in \mathcal{A}$ or $!q(\bar{Z})$ for some
$q \in \mathcal{P}$. Variables in body calls must be bound by the head or by
positive context literals. A selected Layer B rule must be justified by at
least one admissible source: action-schema support, bounded transition evidence,
external sketch binding, trace evidence used only offline, or counterexample
repair evidence. The implementation reports these justifications as proof
objects; schema-only rules may be candidates, but the paper profile treats
unjustified schema action modules as blocking diagnostics.

Layer C contains goal-composer rules. A rule has the form
\[
	+!g : C_g \leftarrow !p(\bar{X}); !g
\]
where $C_g$ mentions read-only goal descriptors and current state facts. These
rules decide which atomic goal should be pursued next for a conjunctive goal.
They do not protect every achieved literal permanently. Temporary facts may be
destroyed, provided the selected rules still make progress toward the final
conjunction inside the validation scope. Generic unsatisfied-goal dispatch rules
are guarded by \texttt{ready\_$p$} contexts derived from the selected support
agenda, so they cannot bypass a selected predecessor goal. This mechanism is a
runtime agenda gate, not a complete goal-order learner by itself; unsupported
goal mutexes and arbitrary reachable mid-state policies remain outside the
current positive claim unless the report validates them explicitly.

\subsection{Candidate Generation}

Candidates are generated without domain-specific code. Primitive candidates come
from \pddl{} action schemas: if an action adds a predicate $p$, the action can
support a Layer B rule for $p$ when its typed parameters and preconditions can be
bound safely. Missing positive preconditions may become calls to other predicate
modules. Delete effects and producer-consumer precondition links contribute
ordering evidence for Layer C.

For typed-overloaded predicates, the generator creates causal-chain module
candidates. This case arises when the same \pddl{} predicate is produced for
different typed argument partitions, such as a movement action producing
\texttt{at(carrier, location)} and an unloading action producing
\texttt{at(package, location)}. If a producer action can establish a
target-action precondition, the candidate first calls the producer predicate
module, then achieves the remaining producible preconditions, and finally
executes the target action. Shared static preconditions, such as a capacity
predecessor relation, may bridge resource variables across the two actions;
mutable resource predicates are not emitted as independent causal-chain goals.
Same-predicate delegation across disjoint typed argument partitions is accepted
only with an explicit type certificate.
The same schema evidence also yields current-resource priority composer rules:
if a target object is already held by a carrier relation used by the target
action, the composer may deliver that object before selecting another goal that
would consume the same carrier resource. This is a bounded resource-ordering
rule, not a complete logistics policy for every reachable mid-state.

External sketch candidates come from verified policy-sketch backends, currently
learner-sketches \cite{Drexler2022LearningSketches}. A learned sketch rule is
accepted only when every condition and qualitative effect has an explicit lifted
binding to \pddl{} literals, primitive actions, predicate subgoals, or read-only
goal descriptors. Object-specific and vocabulary-mismatched features are
rejected unless a principled lifting adapter is provided. The current safe
distance fragment accepts object-free expressions that measure distance from a
unary state predicate to the corresponding read-only goal predicate through a
declared binary role. The compiled effect is a call to the unary predicate goal,
not a replay of training objects.

Trace candidates come from offline evidence only. A planner trace may be used to
identify which action last achieved a target fact, to anti-unify repeated
support patterns, or to explain a counterexample. Such traces are never emitted
as run-time full-trace plans.

\subsection{Feature Binding Contract}

External policies are qualitative feature rules, not executable agent programs.
Before such a rule can enter the selector, every feature condition and effect
must be bound to one of the compiler's safe constructs. The accepted bindings
include unary and binary primitive counts, forward and reverse role counts,
goal-aligned concept and role counts, filtered goal-role counts, nullary
Boolean predicates, and the object-free goal-distance fragment described above.
Each binding records the supported condition operators, effect operators,
required contexts, body calls, action-effect candidates, and rejection reason if
the feature is unsupported.

Concrete object-specific expressions such as \texttt{c\_one\_of(b1)} remain
outside the domain-level claim. Compiling them would copy training objects into
the library and break lifting. Unsupported distance expressions are also
rejected unless their source concept, target goal concept, and binary role can
be recovered as declared \pddl{} predicates with matching arities.

\subsection{Selection Objective}

Candidate selection is encoded as an answer-set optimization problem
\cite{Gebser2018Clingo}. Hard constraints reject unsafe rules, undeclared
symbols, wrong arities, unbound variables, cyclic composer support without a
progress certificate, and candidates unsupported by the selected paper profile.
Additional hard groups require coverage of observed goal-progress transitions
and coverage of bounded reachable non-goal states by at least one applicable
composer rule.

The optimization objective prefers smaller libraries, fewer features, shorter
contexts, fewer body calls, trace- or sketch-justified candidates over
unjustified schema-only candidates, and rules with explicit progress evidence.
The selector therefore chooses the best library within the declared hypothesis
class, not the best possible generalized plan over all languages.

\subsection{Counterexample-Guided Refinement}

Held-out failures are lifted into explicit constraints before the next synthesis
round. A missing atomic module becomes a Layer B atomic-progress constraint when
the failed subgoal is named. A missing \texttt{!g} plan becomes a Layer C
state-coverage constraint. A primitive-action precondition failure becomes a
Layer B repair constraint when the missing precondition is a positive,
producible \pddl{} predicate. If a failure shows that one final goal was
achieved before another causally related goal, it becomes a goal-ordering
constraint for Layer C.

Termination failures are not silently treated as success. Recursive-loop and
step-limit failures keep explicit termination diagnostics, but when the failed
state still contains supported missing positive goals, the system also emits a
companion generative constraint: atomic progress for a named failed predicate
subgoal, or state coverage for a top-level composer failure. Unsupported
negative repairs, undeclared predicates, wrong arities, and unproducible targets
remain structured diagnostics rather than synthesized plans.

\begin{algorithm}[t]
\caption{Synthesize a domain-level lifted \asl{} library}
\label{alg:synthesis}
\begin{algorithmic}[1]
\Require typed \pddl{} domain $D$, training problems $\mathcal{I}_{train}$,
optional external sketches $\mathcal{E}$, held-out problems
$\mathcal{I}_{eval}$
\Ensure lifted \asl{} library $\mathcal{L}$ or structured diagnostics
\State validate the supported \pddl{} fragment of $D$ and $\mathcal{I}_{train}$
\State $\mathcal{I}_{cex} \gets \emptyset$
\Repeat
	\State build schema model from predicates, actions, types, preconditions,
	add effects, and delete effects
	\State collect bounded transition evidence or validated offline trace
	evidence from $\mathcal{I}_{train} \cup \mathcal{I}_{cex}$
	\State generate Layer B candidates from action effects, safe external
	sketches, traces, and repair evidence
	\State generate Layer C candidates from goal descriptors, causal support,
	delete threats, trace ordering, state coverage, and counterexamples
	\State encode safety, coverage, progress, and optimization constraints
	\State $S \gets$ solve the answer-set selection problem
	\If{$S$ is unsatisfiable}
		\State \Return diagnostics explaining the blocking constraints
	\EndIf
	\State derive the selected support agenda and ready-context semantics from
	Layer C rules in $S$
	\State $\mathcal{L} \gets \compile(S)$ into lifted \asl{} plan schemas
	\State run bounded validation and held-out library execution
	\If{all validation and evaluation checks pass}
		\State \Return $\mathcal{L}$ and its provenance report
	\EndIf
	\State lift supported failures into counterexample files or explicit constraints
\Until{the failure is unsupported or no new constraint can be generated}
\State \Return $\mathcal{L}$ with unsupported-failure diagnostics
\end{algorithmic}
\end{algorithm}

\subsection{Compiler Semantics}

Compilation is conservative. A selected Layer B rule becomes an \asl{} plan
whose trigger is the corresponding predicate achievement goal. A selected Layer
C rule becomes a \texttt{+!g} plan. The compiler also records the selected
support agenda used to evaluate \texttt{ready\_$p$} contexts at run time.
Context literals are rendered as an order-independent conjunction. Positive
context literals bind variables before negated literals are checked. Primitive
actions are executed by the same STRIPS-style precondition and effect semantics
used to validate the originating \pddl{} domain.

The compiler rejects any body action or subgoal whose variables are not bound by
the plan head or by positive context literals. It also rejects synthetic
achievement names, grounded plan heads, mutable goal descriptors, undeclared
predicates, and arity mismatches.

\subsection{Guarantee}

\paragraph{\textbf{Proposition 1 (Bounded-Class Soundness).}}
Consider a typed \pddl{} domain and a finite training, counterexample, and
held-out evaluation set inside the supported positive conjunctive fragment.
Assume that every selected Layer B and Layer C rule is generated by the
candidate language above, passes the declared safety checks, and is selected by
the answer-set program under its hard coverage and progress constraints. Assume
also that the compiled library passes bounded validation and held-out execution
under the reported step and depth limits. Then every successful execution trace
reported for the library consists only of declared \pddl{} primitive actions and
declared predicate subgoal calls, contains no run-time full-trace planner call,
and terminates in a state satisfying all positive goal atoms of the evaluated
problem.

\paragraph{\textbf{Proof Sketch.}}
The selector can choose only candidates whose symbols, arities, variables,
goal-descriptor uses, and ready-context uses pass the compiler contract.
Compilation preserves this selected structure: Layer B rules become predicate
achievement plans, and Layer C rules become \texttt{+!g} composer plans over the
recorded support agenda. The executor interprets primitive actions with the same
STRIPS-style precondition and effect semantics used in validation, and
\texttt{ready\_$p$} facts are derived read-only from the selected agenda rather
than asserted as mutable beliefs. Bounded validation and held-out execution then
check the finite set of reported runs directly. Therefore a reported success is
sound for the validated bounded instance set and execution limits.

This guarantee is intentionally conditional. Domains whose solution requires
features outside the safe binding language, concrete object-specific policy
features, unsupported negative or disjunctive goals, or unbounded cyclic-route
ranking without an accepted progress certificate are reported as outside the
current claim rather than being solved by unsafe trace replay.
