\section{Evaluation}
\label{sec:evaluation}

The evaluation has two goals. First, it tests whether external generalized
planners can produce lifted atomic templates that compile into a domain-level
\asl{} library. Second, it tests whether lifted LTLf queries can append
top-level temporal wrapper goals that call that atomic library.

The current formal corpus contains the MOOSE direct train/test domains plus
project-added feature-definable serialized-width domains, organized by the
property coverage in Table~\ref{tab:benchmark-scope}. Each local snapshot
contains:

\begin{verbatim}
src/domains/<domain>/domain.pddl
src/domains/<domain>/train/*.pddl
src/domains/<domain>/test/*.pddl
src/domains/<domain>/source.json
\end{verbatim}

The split is deterministic: MOOSE domains use the official companion
\texttt{training/} and \texttt{testing/} directories; Blocksworld Clear and
Blocksworld On use the KR 2025 learner-policies no-constants train/test
folders; Blocksworld Tower and Depots use the project feature-definable
serialized-width split described in the repository benchmark strategy. The same
split is used by the atomic-template backend selector and by the
query-generation layer.

MOOSE-native parameters follow its paper where applicable. Algorithm~1 uses
three goal permutations by default. Generalized-plan synthesis is evaluated
with a 12-hour, 32-GB budget and five repeated runs, while test-time planning
uses 1800 seconds and 8 GB \cite{Chen2026Moose}. Our runner therefore defaults
to 43200 seconds for MOOSE synthesis, 1800 seconds for MOOSE test-time planning,
and records the random seed for each repetition. The repository safety profile
caps any external learner at 16 GiB, so its synthesis memory ceiling is a
declared reproduction deviation from the paper's 32-GB ceiling. Worker count,
policy-dump timeout, AgentSpeak execution timeout, and VAL timeout are system
parameters of our pipeline rather than MOOSE hyperparameters and are reported
separately.

\subsection{Checks}

Each domain is evaluated through four checks.

\begin{enumerate}
\item Atomic template extraction: the system extracts predicate, arity, and
	polarity templates from the training split.
\item Backend artifact acceptance: the selected backend artifact must parse,
	bind to declared \pddl{} vocabulary, compile to \asl{}, and validate on held
	out instances.
\item Temporal append validation: generated LTLf JSON cases must compile to DFA
	payloads whose relevant transitions contain conjunction and negation only.
\item Library maintenance: each accepted query appends one new top-level
	\texttt{g\_query} goal to the same domain library without introducing
	\texttt{achieve\_*}, \texttt{transition\_*}, or \texttt{dfa\_state} plan
	names.
\end{enumerate}

Classical planners may support external generalized-planning systems, artifact
reproduction, and validation. They are not allowed to replace
the maintained \asl{} library during execution; execution traces are produced
by the generated agent program and then validated externally.

\subsection{Reproducibility}

The reproducibility artifact is the
\path{final_paper_manifest.json} file under \path{paper_artifacts/}. Before submission, the
benchmark registry, generated ASL libraries, LTLf JSON cases, DFA validation
diagnostics, and LaTeX result macros must be regenerated from tracked commands.
Generated experiment outputs live under \path{tmp/paper-final-latest} and are
treated as rebuildable artifacts.
