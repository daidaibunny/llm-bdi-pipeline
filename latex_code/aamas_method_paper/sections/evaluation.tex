\section{Evaluation}
\label{sec:evaluation}

The evaluation separates three questions. First, can the selected library solve
held-out achievement-goal problems without a run-time full-trace planner?
Second, which synthesis mechanisms are necessary on diagnostic splits? Third,
where does the current bounded hypothesis class stop? All reported library rows
execute compiled lifted \asl{} plans with run-time planner flag
\texttt{none}. Classical planners and generalized-planning systems appear only
as offline baselines, evidence providers, or diagnostics.

The main claim is restricted to the first benchmark class in
Table~\ref{tab:domain-families}: positive conjunctive achievement goals whose
strategy is a compiled feature-definable modular sketch. A row is part of the
strict paper profile only when an external learned sketch rule is parsed,
feature-bound, selected by the answer-set program, compiled to a non-empty
executable body, and validated without run-time full-trace planning. Rows that
test mechanism necessity without an external learned sketch are mechanism-stress
rows. Rows that exercise trace evidence, resource movement, route ranking, or
unsupported fragments are boundary rows rather than main coverage claims.

Table~\ref{tab:main-results} shows the strict-profile positive evidence. The
three Blocksworld rows use the verified \texttt{blocks\_4\_on\_2}
learner-sketches artifact and the strict paper profile. The learned library
solves the original first-20 split and two larger satisfiable held-out splits.
The large-tower split contains towers from 50 to 140 blocks. The mixed split
contains nontrivial initial towers and multiple final towers with both
\texttt{ontable} and \texttt{on} goals.
These rows are the current main claim because Blocksworld is the standard
construction benchmark where prior work gives feature-definable policies for
\texttt{clear}, \texttt{on}, and full goal configurations
\cite{Frances2021LearningGeneralPolicies}. The result is therefore not a claim
that every domain listed in Table~\ref{tab:domain-families} is already
supported. To match the scale of prior generalized-planning papers, the final
evaluation target is three property-defined classes with roughly nine standard
domains or problem classes: Gripper, Ferry, and Miconic for the goal-separable
or serialisable class; Spanner, Childsnack, Barman, Visitall, and Delivery for
the bounded-width sketchable subgoal-structure class; and Qclear, Qon, and full
Blocksworld for the feature-definable construction class. These domains become
claimed coverage only after their own compiled \asl{} libraries pass the same
strict protocol. LTLf temporal extended goals are evaluated as overlays on
representatives of these classes, not as a fourth benchmark class.

Table~\ref{tab:mechanism-stress} is intentionally interpreted narrowly. The
mechanism evidence for Layer C and counterexample refinement comes from
Labworkflow: the full refinement row solves five held-out dependent problems,
while removing either Layer C ordering or counterexample refinement solves none
of them; the reports record missing composer plans for the reagent-log goal.
The stress domain is constructed so that one goal consumes a resource needed by
another final goal: an analysis action can consume a reagent, while the
reagent-log goal still requires that reagent to be available. A valid composer
must therefore learn to schedule the logging goal before the consuming analysis
goal. This is the precise behavior tested by the full, no-Layer-C, and
no-refinement rows.
The Blocksworld no-sketch row remains at
\ResultNoExternalSketchFirstTwoZeroCoveragePercent{} on first20, so it is not
used as evidence that external sketches are necessary on that split. It shows
that schema-derived modules can already cover this small set.

The refinement rows test the intended behavior of the counterexample loop:
failures are not merely logged. Supported missing-module, state-coverage,
goal-ordering, and repair failures become selector-consumable constraints in
the next synthesis round. Pure lifted goal-ordering failures are consumed as
explicit Layer C constraints without forcing the failed held-out problem into
bounded transition-system exploration. Termination failures remain diagnostics,
but the
implementation also emits atomic-progress or state-coverage companion
constraints when the missing goal is inside the supported fragment.

Transport and Marsrover demonstrate broader \pddl{} parsing, typed action
binding, route/resource modules, and validated offline trace evidence. They are
reported as boundary rows because their current reports are not strict
paper-profile claims. In particular, they validate execution from problem
initial states, while arbitrary reachable mid-state policies remain outside the
main claim unless a future report validates that stronger setting. This boundary
matches the literature: route-heavy domains require distance, ranking, or
transitive-closure features, and MOOSE similarly removes path-finding components
from some benchmark families when testing goal-regression assumptions
\cite{Chen2026Moose}.

Feature-binding diagnostics are part of the evaluation artifact. Safe lifted
features, including object-free goal-distance features, can enter the selector.
Concrete object-specific features and unsupported distance expressions remain
structured rejections. This distinction is important because the method's output
is a lifted domain-level library; copying training objects into \asl{} would
turn the result into a problem-specific artifact.

\subsection{Reproducibility}

The reproducibility artifact is specified by the tracked final-paper manifest
under \path{paper_artifacts/}. The manifest records the generation and
validation commands, required repository and external inputs, matrix
configurations, expected library and baseline rows, result counts, and resource
policy. Generated data live under \path{tmp/paper-final-latest} and are treated
as rebuildable outputs, not as the source of truth. Each generation run copies
the manifest into the output directory; validation checks that this copy matches
the tracked manifest, that all matrix configurations and table rows match the
manifest contract, that result macros equal \path{comparison.json}, and that all
library rows use no run-time full-trace planner. Backend pins and resource
limits are part of the same machine-checked contract: validation checks the
local commits for learner-sketches, h-policy-learner, d2l, MOOSE, the MOOSE
benchmark submodule, and VAL when those repositories are declared as pinned
inputs. External generalized-planning learners must be run through a memory
guard, and the manifest records the preflight commands that print the guarded
paper-configuration runs. The main verification commands are the manifest's
generate/validate commands, \path{uv run pytest -q}, and
\path{latexmk -pdf main.tex}.
