\section{Evaluation}
\label{sec:evaluation}

The evaluation separates three questions. First, can the selected library solve
held-out achievement-goal problems without a run-time full-trace planner?
Second, which synthesis mechanisms are necessary on diagnostic splits? Third,
where does the current bounded hypothesis class stop? All reported library rows
execute compiled lifted \asl{} plans with run-time planner flag
\texttt{none}. Classical planners and generalized-planning systems appear only
as offline baselines, evidence providers, or diagnostics.

The main claim is restricted to positive conjunctive achievement goals. A row is
part of the strict paper profile only when an external learned sketch rule is
parsed, feature-bound, selected by the answer-set program, compiled to a
non-empty executable body, and validated without run-time full-trace planning.
Rows that exercise trace evidence, resource movement, or larger unsupported
fragments are reported as boundary rows rather than main coverage claims.

Table~\ref{tab:main-results} shows the main positive evidence. The three
Blocksworld rows use the verified \texttt{blocks\_4\_on\_2} learner-sketches
artifact and the strict paper profile. The learned library solves the original
first-20 split and two larger satisfiable held-out splits. The large-tower split
contains towers from 50 to 140 blocks. The mixed split contains nontrivial
initial towers and multiple final towers with both \texttt{ontable} and
\texttt{on} goals. Labworkflow is smaller, but it exercises the
counterexample-refinement path: a held-out failure is lifted into a new composer
constraint and the next library solves both problems.

The ablation table is intentionally interpreted narrowly. The Blocksworld
no-sketch row remains at
\ResultNoExternalSketchFirstTwoZeroCoveragePercent{} on first20, so it is not used
as evidence that external sketches are necessary on that split. It shows that
schema-derived modules can already cover this small set. The mechanism evidence
for Layer C and counterexample refinement comes from Labworkflow, where removing
either mechanism leaves one of two problems unsolved with the diagnostic
\texttt{no applicable plan for !reagent\_logged(r1)}.

The refinement rows test the intended behavior of the counterexample loop:
failures are not merely logged. Supported missing-module, state-coverage,
goal-ordering, and repair failures become selector-consumable constraints in
the next synthesis round. Termination failures remain diagnostics, but the
implementation also emits atomic-progress or state-coverage companion
constraints when the missing goal is inside the supported fragment.

The raw learner-sketches row is an artifact audit, not an executable baseline:

are useful for comparison, but they do not produce reusable \asl{} libraries.

Transport and Marsrover demonstrate broader \pddl{} parsing, typed action
binding, route/resource modules, and validated offline trace evidence. They are
reported as boundary rows because their current reports are not strict
paper-profile claims. In particular, they validate execution from problem
initial states, while arbitrary reachable mid-state policies remain outside the
main claim unless a future report validates that stronger setting.

Feature-binding diagnostics are part of the evaluation artifact. Safe lifted
features, including object-free goal-distance features, can enter the selector.
Concrete object-specific features and unsupported distance expressions remain
structured rejections. This distinction is important because the method's output
is a lifted domain-level library; copying training objects into \asl{} would
turn the result into a problem-specific artifact.

\subsection{Reproducibility}

The final data package is generated by the final-data script in
\path{scripts/}. It writes fixed matrix configurations, backend-audit logs,
completed baseline records, library reports, \path{comparison.json}, and result
macros in the paper's \path{generated/} directory. The fixed package used here
is under \path{tmp/paper-final-latest}. Backend pins and resource limits are
recorded in the project instructions; external generalized-planning learners
must be run through a memory guard, and final library evaluation uses no
run-time full-trace planner. The verification commands are
\path{uv run pytest -q} and \path{latexmk -pdf main.tex}.
